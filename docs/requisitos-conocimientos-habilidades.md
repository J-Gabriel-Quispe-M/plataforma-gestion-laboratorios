---

## 1. Diagnóstico de la Propuesta Base

La propuesta base reconoce correctamente que cada rol necesita conocimientos específicos. Sin embargo, presenta los siguientes aspectos por mejorar:

- **Requisitos elevados para la primera fase:** Se solicitan conocimientos avanzados en Kubernetes, GitOps, seguridad y arquitectura a estudiantes que todavía se encuentran en formación.
- **Niveles poco verificables:** Términos como "intermedio", "avanzado" o "senior" no indican qué tareas debe ser capaz de realizar cada participante.
- **Falta de diferenciación:** No se distingue entre conocimientos obligatorios de ingreso y habilidades que se aprenderán durante el proyecto.
- **Ausencia de un plan de capacitación:** No se establecen talleres, mentorías ni mecanismos para cerrar las brechas de conocimiento.
- **Riesgo de dependencia:** Las actividades críticas podrían quedar concentradas en pocos integrantes con experiencia técnica.

---

## 2. Clasificación Propuesta de Competencias

Se propone organizar los requisitos en tres categorías:

1. **Conocimientos de ingreso:** Competencias mínimas que el participante debe tener antes de asumir el rol.
2. **Conocimientos por desarrollar:** Habilidades que se adquirirán durante el proyecto mediante prácticas y capacitación.
3. **Conocimientos especializados:** Competencias avanzadas destinadas a responsables técnicos, mentores o fases posteriores.

Esta clasificación permite incorporar estudiantes con distintos niveles de experiencia sin comprometer la calidad del proyecto.

---

## 3. Requisitos Propuestos por Rol

| Rol | Conocimientos de ingreso | Conocimientos por desarrollar | Apoyo requerido |
|---|---|---|---|
| Alumno desarrollador | Programación básica, Git y fundamentos de bases de datos | API REST, pruebas, Docker y seguridad básica | Talleres y revisión docente |
| Alumno DevOps | Linux, redes básicas y Git | CI/CD, Docker Compose, registros de imágenes y automatización | Prácticas guiadas y mentoría |
| Responsable de procesos | Documentación y trabajo colaborativo | BPMN, matriz RACI e indicadores de desempeño | Asesoría del profesor |
| Profesor o mentor | Experiencia en el área asignada | Coordinación, seguimiento y transferencia de conocimiento | Reuniones periódicas |
| Encargado de laboratorio | Gestión de equipos y normas institucionales | Inventario digital, reservas, incidencias y trazabilidad | Capacitación funcional |
| Director o responsable institucional | Conocimiento de las necesidades de la institución | Supervisión mediante reportes e indicadores | Informes periódicos |

---

## 4. Proceso de Evaluación y Asignación de Roles

Antes de formar los equipos se recomienda aplicar el siguiente proceso:

1. Registrar los conocimientos, experiencia, intereses y disponibilidad de cada participante.
2. Realizar una evaluación diagnóstica breve sobre programación, Git, bases de datos, Linux y trabajo colaborativo.
3. Asignar los roles considerando las competencias demostradas y no solamente el semestre o promedio académico.
4. Designar un responsable principal y una persona de apoyo para cada actividad crítica.
5. Revisar la asignación al finalizar cada fase y realizar cambios cuando sea necesario.

---

## 5. Plan de Capacitación Progresiva

Se propone desarrollar las siguientes actividades antes y durante la primera fase:

- Taller de Git, ramas y trabajo colaborativo.
- Introducción a Docker y Docker Compose.
- Fundamentos de bases de datos y desarrollo de API REST.
- Capacitación en documentación de procesos, BPMN y matriz RACI.
- Introducción a seguridad, licencias de software y trazabilidad.
- Sesiones de mentoría con profesores o estudiantes de mayor experiencia.
- Revisión periódica de avances y dificultades por cada Chapter.

Las capacitaciones deben incluir ejercicios verificables. Por ejemplo, un alumno podrá demostrar su manejo de Git mediante la creación de una rama y un pull request, y su conocimiento de Docker mediante la construcción y ejecución de una imagen sencilla.

---

## 6. Implementación por Fases

### Fase universitaria

Se priorizarán competencias fundamentales y herramientas accesibles:

- Git y GitHub.
- Programación y API REST.
- PostgreSQL.
- Docker y Docker Compose.
- Pruebas básicas.
- Documentación y trabajo ágil.

### Fase avanzada o empresarial

Después de validar el funcionamiento básico podrán incorporarse:

- Kubernetes.
- GitOps.
- Harbor y escaneo de imágenes.
- Automatización con Terraform y Ansible.
- Seguridad avanzada y observabilidad.

Esta implementación progresiva reduce la complejidad inicial y permite que los participantes adquieran experiencia antes de asumir responsabilidades especializadas.

---

## 7. Criterios Verificables

Para evitar descripciones subjetivas, cada competencia debe relacionarse con una evidencia:

| Competencia | Evidencia sugerida |
|---|---|
| Git básico | Crear una rama, registrar commits y abrir un pull request |
| Programación | Implementar una funcionalidad pequeña siguiendo estándares del equipo |
| Bases de datos | Diseñar tablas y realizar operaciones de consulta y actualización |
| Docker | Construir y ejecutar una imagen con su documentación |
| Procesos | Elaborar un flujo BPMN y una matriz RACI |
| Seguridad básica | Identificar vulnerabilidades y aplicar medidas de corrección |
| Trabajo colaborativo | Participar en revisiones, reuniones y documentación del equipo |

---

## 8. Resultados Esperados

La propuesta permitirá:

- Asignar responsabilidades de manera objetiva.
- Incluir estudiantes con distintos niveles de experiencia.
- Reducir los riesgos producidos por la falta de conocimientos.
- Promover un aprendizaje progresivo con acompañamiento docente.
- Evitar que el conocimiento crítico dependa de una sola persona.
- Preparar a los participantes para las fases avanzadas del proyecto.

---

## 9. Conclusión

Definir requisitos por rol es necesario para organizar correctamente la gestión y el desarrollo de la plataforma. No obstante, estos requisitos deben ser alcanzables, verificables y coherentes con el carácter formativo de una primera fase universitaria. La capacitación, la mentoría y la evaluación progresiva deben formar parte de la organización del proyecto.
